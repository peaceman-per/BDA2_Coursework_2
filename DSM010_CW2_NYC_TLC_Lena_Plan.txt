===========================================================
B) Project concept (what the notebooks implement)
===========================================================

Topic title (suggested):
"City-scale mobility intelligence from NYC TLC trips (2018–2025): demand dynamics, reliability, and forecasting using distributed Spark on Lena"

Core questions/hypotheses (to embed in notebook/report):
H1: Pickup demand shows strong zone-hour seasonality and differs significantly across Yellow vs Green vs FHV.
H2: Trip duration reliability (e.g., p95 duration) varies by zone and time and shifts across years (structural changes).
H3: A time-aware Spark ML model (GBTR) can forecast next-hour pickups per zone better than a seasonal naive baseline.
H4: Partitioning/caching choices measurably affect runtime for large aggregations (technical merit).

ML focus:
- Forecast next-hour pickups per zone-hour (regression) with lag features and time-based evaluation.
(Alternative optional extension: anomaly detection using residuals and store anomaly "case files".)

Technology usage justification:
- HDFS: storage of multi-year raw Parquet + curated Parquet marts.
- Spark/PySpark: distributed ETL + structured analytics + ML pipelines.
- PostgreSQL: curated "gold" marts for reproducible ad-hoc queries and report tables; store model metrics and experiment logs.

SECTION 2 — Data discovery & ingestion (Code)
Goal: robust loading across many partitions without manual month list edits.

Implementation:
- Build a month iterator from START_YM to END_YM.
- For each service_type and month:
  - expected filename: f"{service}_tripdata_{YYYY}_{MM}.parquet"
  - expected location patterns:
    a) {HDFS_INPUT_ROOT}/{YYYY}/{filename}
    b) {HDFS_INPUT_ROOT}/{YYYY}/{service}/{filename} (if present)
  - Try read; if missing, log and continue (but count missing files).
- Union by name with allowMissingColumns=True (Spark 3+).
- Add columns:
  - service_type (string)
  - year, month (int) extracted from file name or pickup_datetime
- Print counts per service_type.

Acceptance:
- Produces raw DataFrames for each service and a unified raw union.

SECTION 3 — Schema harmonisation ("Silver") (Code)
Problem: Yellow/Green/FHV have different columns and names across years.

Approach:
- Define a canonical schema with columns used for analytics + ML:
  Required canonical columns (minimum):
  - service_type
  - pickup_datetime (timestamp)
  - dropoff_datetime (timestamp) (if present; for FHV may differ)
  - pu_location_id (int)
  - do_location_id (int)
  - passenger_count (int, nullable)
  - trip_distance (double, nullable)
  - total_amount (double, nullable) (or derived)
  - fare_amount (double, nullable)
  - year, month, day, hour, dow (derived)
  Derived:
  - trip_duration_sec = (dropoff - pickup)
  Quality flags:
  - valid_time (duration >0 and < some threshold, e.g., 4 hours)
  - valid_locations (PU/DO not null)
- Implement per-service mapping function:
  standardize_yellow(df)
  standardize_green(df)
  standardize_fhv(df)

Cleaning:
- Filter out invalid pickup_datetime nulls.
- Filter duration outliers for reliability analysis (keep raw counts too).
- Coalesce location ids to int.

Write Silver:
- df_silver.write.mode("overwrite").partitionBy("service_type","year","month").parquet(SILVER_PATH)

Acceptance:
- A single df_silver with canonical columns exists.
- Silver Parquet saved to HDFS and re-loadable.

SECTION 6 — ML dataset preparation (time-series aware) (Code)
Goal: forecast next-hour pickups per zone (regression).

Steps:
1) Start from zone_hour_demand (preferably a single service_type or combined):
   Option A (simpler + strong): model Yellow only for quality, then compare to Green/FHV descriptively.
   Option B: include service_type as categorical feature; model all combined.
2) Create features using window functions:
   - lag_1 = pickups at t-1 hour
   - lag_24
   - lag_168 (weekly)
   - rolling_mean_24
   - rolling_mean_168
   - hour_of_day, day_of_week, month
3) Define label:
   - y = pickups at t+1 hour (lead)
4) Drop rows with null lags/leads (edges).

Scaling strategy:
- Optionally restrict to TOP_ZONES_N zones by total pickups to reduce training time while keeping big-data credibility.
- Keep the full aggregated mart for analytics; use top zones for modelling.

Acceptance:
- model_df with columns [features..., label] exists.
- Model dataset sizes printed.

SECTION 7 — Baseline vs Spark MLlib pipeline (Code)
Baseline:
- seasonal naive: y_hat = lag_24 (or lag_168) and compute MAE/RMSE.

Spark ML pipeline:
- If categorical: StringIndexer + OneHotEncoder for service_type and optionally zone_id.
- VectorAssembler for numeric + one-hot.
- Model: GBTRegressor (or RandomForestRegressor as alternative).
- Param tuning:
  - CrossValidator with modest grid:
    - maxDepth in [3,5]
    - maxIter in [20,50]
    - stepSize maybe [0.05, 0.1]
- Time-series split:
  - Train: 2018-01 to 2024-12
  - Test: 2025-01 to 2025-11
  (No randomSplit.)

Evaluation:
- RegressionEvaluator MAE and RMSE.
- Report improvement over baseline.

Acceptance:
- Metrics table exists:
  - baseline_mae, model_mae, baseline_rmse, model_rmse
  - train/test date ranges
  - model params
- Sample predictions saved.
